# Kill Team 战棋助手 — UX 设计首版草案

本草案把 PRD 中定稿的产品意图（无后端、单屏全公开、pass-and-play、push 式推进、一击交互、先验拦截、可审计）转译成可直接进入视觉/组件阶段的 IA、布局、流程与交互规范。读者：构建者本人（Winst）及下游 Epic/Story 拆分。

草案中所有 `FR-x` 引用回指 PRD 的功能需求；`[UX-OQ-x]` 标记留给后续的开放 UX 问题（见 §11）。

---

## 1. 设计原则与基调

按重要性排序，冲突时上位优先。

1. **桌上易读优先**。设备横放在两玩家之间、距离 ~50–80cm、光线不一。字号、对比、图标尺寸按「桌面阅读」而非「办公阅读」定。可读性 > 信息密度 > 美观。
2. **减少操作步数（一击交互为纲）**。点目标即开始结算，不在弹窗/子页间来回（FR-13）。每个确认按钮都需回答：「这步能否由引擎自动推进而仍然可信？」若能则自动。
3. **触控友好**。平板为主要场景之一。最小可点击目标 44×44px；关键操作（投骰、确认伤亡、回滚）加大到 56px。可点击区与可读区分离明确，避免误触。
4. **可信即采纳**。自动结算要让玩家「想用而非想查」：默认把引擎结果当主结果呈现，但每一步都附可展开依据（FR-16/17）。怀疑是低频路径，不能阻塞高频路径。
5. **单屏全公开，不藏不隐**（FR-18）。任何一方信息对双方同时可见；仅以高亮区分「当前主动玩家」，绝不靠遮罩/折叠藏信息。
6. **降级要可见**。规则缺口/人工裁定状态必须显式呈现（护栏），绝不静默猜测（D-17/OQ-4）。
7. **刷新即重置作为已知取舍**。不做存档界面、不做续用入口、不做「未保存提醒」弹窗（D-20/FR-19 已移除）。仅在对局进行中给出一次性、非阻塞的「刷新将丢失本局」浏览器原生 `beforeunload` 兜底。

视觉基调（建议，非定稿）：深色中性底（降低桌面反光）+ 双阵营分色（暖/冷）做敌我区分；强调色只给「需立即关注的状态」（待激活、待确认结算、人工裁定、effect 到期）；结算流水线用等宽/紧凑列表呈现，不用炫技动画。动画仅用于状态切换的因果说明（如伤害应用），不用于装饰。

## 2. 信息架构（IA）

### 2.1 顶层导航

三个顶层入口，无存档面（D-20）。

```
Kill Team 战棋助手
├── ① 建队 (Roster)        — 阵营/特工/装备组装 + 实时合法性校验 (FR-20)
├── ② 对局 (Match) ★核心   — 棋盘 + 单位 + 状态 + 结算流水线 + 日志回放
└── ③ 规则查询 (Rules)     — 关键词/effect/武器/阵营机制全文检索 (FR-23)
```

**进入规则**：
- 启动直达「建队」或「对局」入口选择页（不强制教程，Non-Goal 不做教学层）。
- 「对局」需要一个双方建好的阵容才能进入；空阵容时入口置灰并提示「先建队」。
- 「规则查询」可从任意位置顶栏一键进入，且支持从结算流水线内联跳入（见 §7）。
- 无「设置」顶层入口（v1 配置面最小）；骰源（电子/物理）切换放在对局内的流水线区，就近操作。

### 2.2 对局面内部的二级结构

对局界面是单页应用（不切顶层路由），内部用**区域划分**而非页面切换组织：

```
对局面
├── 棋盘区 (Board)              — 自由坐标棋盘，特工/地形/目标点/几何可视化
├── 状态带 (Status Strip)       — 回合·阶段·主动玩家·CP/AP·VP 提示
├── 单位面板 (Operatives)       — 双方特工卡片列表 + 受创/effect/命令状态
├── 行动/激活指挥 (Action Bar)  — push 式当前该做什么 + 确认控件
├── 结算流水线 (Pipeline)       — 一击交互展开的伤害流水线（折叠态/展开态）
└── 日志回放 (Log)              — 可审计历史 + 单步回滚
```

「单屏全公开」的组织方式：**全部区域同屏可见，无 Tab 切换、无模态遮挡结算**。模态只用于真正需要专注的子任务（如物理骰录入、人工裁定确认、规则原文浮层），且都可一键关闭回到全公开态。

**分区配比**：棋盘区约占左 55–60%（最大信息载体），中间 22% 给「状态带 + 行动指挥 + 单位面板」，右侧 18% 给「流水线 + 日志回放」。三栏可在 1024–1280px 之间压缩流水线宽度到图标条；<1024px 切换到平板布局（见 §3.2）。

### 3.2 平板版线框（768–1024px，横屏，触控）

平板是 pass-and-play 共用的主要场景。策略：**棋盘置上、控制置下**，下方控制区用大触控目标；流水线/日志在结算时从底部上滑为**半屏抽屉**，不打断棋盘视野。

```
┌─────────────────────────────────────────────────────────┐
│ [KT] 对局 · 规则        TP2/4 交战  ●死亡天使  [电子|物理] │
├─────────────────────────────────────────────────────────┤
│                                                         │
│                     棋 盘 区 (上 ~55%)                   │
│              (触控拖放特工，双指缩放/平移)                 │
│                                                         │
├─────────────────────────────────────────────────────────┤
│ ▼ 当前该做: 激活一名就绪特工      [查看就绪] [反应] [结束] │ ← 大按钮 56px
├─────────────────────────────────────────────────────────┤
│ ┌─主动:死亡天使─┐ ┌─对手:军团兵─┐                       │
│ │▣步兵A 就绪     │ │▣恐虐兵 待机  │  ← 单位面板(横向滚动) │
│ │▣步兵B 受伤待机 │ │▣奸奇术 ⚠毒   │     (下 ~25%)        │
│ │▣队长 ●激活     │ │▣...          │                      │
│ └────────────────┘ └──────────────┘                      │
├─────────────────────────────────────────────────────────┤
│ 〔结算/日志抽屉手柄〕▲ 上滑展开流水线与历史回放           │ ← 结算时自动上滑
└─────────────────────────────────────────────────────────┘
```

**平板关键调整**：
- 单位面板改为横向滚动卡片（手指横滑比纵向更适合对面对坐）。
- 行动指挥区（push 提示 + 确认）固定在棋盘正下方，保证两位玩家都能看到「现在该谁」。
- 流水线默认折叠为底部手柄；触发结算时自动上滑到约 45% 屏高，露出步骤列表与「投骰/确认」大按钮；拖到顶部可全屏（密集审计场景）。
- 主动玩家用色块横条 + 文字双编码，色盲安全。

### 3.3 单屏全公开的组织要点

- 双方特工在「单位面板」中**并列**呈现（桌面：上下分区；平板：左右并排），不为对手设隐藏态。
- 对手的 effect、命令、CP/AP、装备**全部明示**；只有「当前主动玩家可操作」这个权限位在变，不藏信息。
- 切换主动玩家时用整屏顶部一条 2s 的色带横扫 + 状态带文字更新，不做模态「轮到 XX」弹窗（避免打断节奏）。

## 4. 核心流程 1 —「攻击该目标」一击交互

这是产品根价值（UJ-1）的交互落点。设计目标：**从点目标到伤亡，零切屏、零模态、零多余确认**。

### 4.1 流程图

```
[棋盘上点选攻击方特工]   ← 必须先有一个「激活中/可行动」的特工被选中
         │
         ▼
[点选目标特工]
         │
         ▼ ┌──────── FR-10 资格判定（先验，不合法即拦，见 §6）────────┐
         │ │  命令(交战/隐匿)? · 可见? · 掩护? · 遮挡? · 射程? ·     │
         │ │  控制范围内无己方? · 同激活未重复? · 武器资格?           │
         │ └───────────────────────────────────────────────────────┘
         │
    ┌────┴─────┐
    ▼ 不合法   ▼ 合法
[拦截卡]      [自动判定: 射击 or 近战?]
   │              │  ← 据双方距离 vs 控制范围；两者皆可则给二选一 chips
   │              ▼
   │         [结算流水线展开]  ◀── 右侧/底部抽屉，棋盘保持可见
   │              │
   │              ▼
   │         [投骰步骤]
   │         ├ 电子骰: [投!] 按钮 → 动画 → 数值
   │         └ 物理骰: [手动录入] → 网格勾选每颗 D6 结果 → [确认]
   │              │  ← 两种骰源进入同一后续流水线（FR-3）
   │              ▼
   │         [流水线分步执行，每步: 输入→修正→输出，附依据▾]
   │         ①前置检查 ②攻击骰 ③命中修正 ④保留 ⑤重掷/升级
   │         ⑥防御骰 ⑦穿刺/掩护豁免 ⑧抵挡分配 ⑨造伤 ⑩减免+伤亡
   │              │  ← 每步默认折叠结论行，点击 ▾ 展开依据
   │              ▼
   │         [确认伤亡 ▶]   ← 唯一一次主动确认（可审计 FR-16/17）
   │              │
   │              ▼
   │         [状态自动更新] 耐伤↓ / 受伤受创阈值 / effect 到期 / 残废移除
   │              │
   └──────────────┴──→ [流水线收起，回到棋盘，push 推进下一步]
```

### 4.2 关键屏描述

**屏 A — 选目标后的资格反馈（不合法）**：
目标头像变红框 + 一行「无法攻击」+ 缺哪条的清单（每条带规则原文链接）。无确认按钮（无可确认之事），只给「查看原因」与「取消」。对应 §6。

**屏 B — 选目标后的资格反馈（合法，含射击/近战歧义）**：
目标头像绿框，状态带下方出现 chips：`[射击 ▸] [近战 ▸]`（仅当两者皆合法）。点其一即进入对应流水线，不弹模态。仅一种合法时跳过 chips，直接进入。

**屏 C — 流水线展开态（射击示例）**：
```
射击结算 #3   步兵A(交战) → 纳垢兵(无掩护)        [物理骰 ⇄ 电子骰]
─────────────────────────────────────────────────────────
①前置检查  ✓ 有效目标·无遮挡·无己方遮挡          [▾依据]
②攻击骰   4 颗 D6 = [2,5,1,6]                    [▾依据][重投?]
③命中修正 残暴+1关键 / 受创无(命中-1不与受创叠)  [▾依据]
④保留     普通:5  关键:6   自然1必败丢弃1        [▾依据]
⑤防御骰   3 颗 D6 = [3,4,2]  穿刺-1(关键触发)     [▾依据]
⑥掩护豁免 无(无掩护)                              [▾依据]
⑦抵挡分配 关键(暴击)无法被普通格挡 → 1 伤         [▾依据]
⑧减免     恼人生命力(纳垢) ≥3普通 D6=5≥5 减1 → 2 伤 [▾依据]
⑨伤亡     纳垢兵 耐伤 13→11                      [▾依据]
─────────────────────────────────────────────────────────
                          [◀ 回滚此步]  [确认伤亡 ▶]
```

要点：
- 默认每步只显示**结论行**（如「保留: 普通1 关键1」），怀疑者点 `[▾依据]` 展开规则原文 + 修正来源链（FR-17）。
- 每步可单独回滚（FR-16），不必整条重来——例如投骰后想改物理录入值。
- 「确认伤亡」是流水线唯一的强制确认点，落实可审计 + 误操作兜底。
- 流水线运行期间棋盘**保持可见**（桌面右侧抽屉；平板底部半屏），玩家能看到攻击方/目标在棋盘上的位置。

**屏 D — 物理骰录入**：
进入投骰步骤且骰源为「物理」时，弹出轻量浮层（非全屏模态）：
```
   手动录入攻击骰 (4 颗 D6)
   ┌───┬───┬───┬───┬───┬───┐
   │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │
   ├───┼───┼───┼───┼───┼───┤
   │ ×1│   │   │×1 │   │ ×2│   ← 点击格子累加该面数量
   └───┴───┴───┴───┴───┴───┘
   已录: 1,4,6,6   总数 4/4 ✓
                       [取消]  [确认 ▶]
```
格子点击累加、再点减一，总数达标才允许确认。规避键盘输入（平板触控）。

## 5. 核心流程 2 — push 式回合推进

引擎主动提示「现在该做什么」，玩家确认。对应 FR-15。push 的落点是**行动指挥区（Action Bar）**——固定可见的一条带状区，永远回答「现在该谁、做什么」。

### 5.1 push 提示的状态机映射

```
部署阶段 (双方降落区交替部署特工 FR-24)  push:「轮到 X — 在降落区放置特工」
   │
   ▼ 转折点开始
   ▼ push: 「战略阶段 — 双方选计谋(耗CP)」
   │       控件: [我方计谋 ▸] [对手计谋 ▸] [战略阶段结束 ▶]
   ▼
交战阶段
   │
   ▼ push: 「轮到 死亡天使 — 激活一名就绪特工」
   │       控件: [查看就绪 ▸]  (棋盘上就绪特工高亮脉动)
   ▼ 选定特工
   push: 「选命令: 交战 / 隐匿」 → [交战][隐匿]
   ▼
   push: 「行动(AP 2/2): 选行动」
   │       [射击][近战][转移][冲锋][后撤][-rise]... 非法项置灰(见 §6)
   ▼ 行动后
   push: 「AP 1/2 剩余 — 继续行动 或 [结束激活 ▶]」
   ▼ 激活结束
   push: 「翻待机 — 轮到对手激活」  (整屏色带切换)
   ▼ 双方全待机
   push: 「所有特工待机 — [反应阶段? / 结束转折点 ▶]」
   ▼
下一转折点 (结算目标点控制 + VP 得分 FR-25/26) / 战斗结束 → VP 总分判胜负 (FR-27)
```

### 5.2 push 控件规范

- **永远显示一个主操作大按钮**（当前最可能的下一步），辅以 1–3 个次操作。避免「空白选择」状态。
- 当前 push 提示用主动玩家的阵营色描边；切换主动玩家时整条色带换色。
- 玩家可点 `[查看就绪]` / `[查看可行动]` 让棋盘上合规特工**脉动高亮**，降低寻找成本。
- 任何「结束」类按钮（结束激活、结束转折点）需在带内显示「之后会发生什么」的微提示（如「结束转折点 → effect 到期 ×N、CP +2」），避免误点。
- 反应（Reaction）触发时（对方全待机、满足条件），push 带临时切到「反应机会 — [反应 ▸][放弃]」，醒目但不抢模态。

## 6. 先验合法性拦截 UX

对应 FR-14 / FR-12。原则：**不合法时立即拦在源头，用「缺什么」清单替代「错误」模糊提示**。

### 6.1 拦截触发点与呈现

| 触发场景 | 缺什么（示例） | 拦截呈现 |
|---|---|---|
| 点目标想射击 | 目标隐匿且有掩护 | 目标红框 + 「无法射击：目标隐匿且有掩护（隐匿+掩护不可为目标）[规则▸]」 |
| 选射击行动 | 攻击方在控制范围内 | 行动按钮置灰 + tooltip「控制范围内禁射击 [规则▸]」 |
| 选冲锋 | AP 不足 | 行动按钮置灰 + 「AP 不足（需1，剩0）」 |
| 后撤后想转移 | 后撤激活已禁转移 | 转移按钮置灰 + 「后撤后该激活禁转移/冲锋 [规则▸]」 |
| 选近战 | 控制范围内无敌方 | 行动按钮置灰 + 「近战需敌方在控制范围内」 |
| 结算中武器资格不符 | 武器关键词不匹配 | 流水线前置检查红字 + 不进入投骰 |

### 6.2 拦截卡（Interceptor Card）规范

不合法时弹出的**非阻塞卡片**（不遮棋盘，可一键关）：

```
┌──────────────────────────────────────┐
│ ✕  无法执行: 射击                      │
│ ─────────────────────────────────── │
│ 缺少资格条件:                         │
│  • 目标处于「隐匿」命令               │
│  • 且目标在掩护中                     │
│  → 隐匿+掩护时不可作为目标            │
│ ─────────────────────────────────── │
│ [查看规则要点 ▸]   [知道了]           │
└──────────────────────────────────────┘
```

要点：
- 列**每条**缺失条件，不笼统说「不合法」。
- 每条可链到规则查询（FR-23）的原文条目。
- 知道后不强制清除棋盘选择（玩家可能想换目标），只解除当前意图。
- 对「行动限制类」（如后撤后禁冲锋）采用**按钮置灰 + tooltip** 而非卡片，因为意图尚未成形。

## 7. 棋盘交互

对应 FR-8（自由坐标）/ FR-9（几何计算）。自由坐标意味着位置全靠玩家手动维护，UX 必须让「放/移/标」这三件事极低成本。

### 7.1 放置与移动特工

- **拖放放置**：从单位面板的特工卡片拖到棋盘任意位置即落子；底座圆圈按阵营色描边。
- **拖动移位**：棋盘上长按（触控）/按下拖动（鼠标）特工即可移动；松手落定。移动时实时显示与起点的英寸数（用于回合内合规检查，如「转移 ≤移动值」）。
- **朝向**：特工头像顶部小三角指示朝向；双击旋转 45°（朝向影响 LOS 头部射线起点）。
- **批量**：建队完成后进入对局时，提供「按阵型一键摆放」辅助（非强制），减少初始摆盘成本。

### 7.2 地形与掩护标记

- **预设地图模板**：开局选模板一键载入地形/目标点/降落区（随包静态数据，D-28）；物理板非标准时再用「画地形」手画（会话内，刷新重置）。
- **地形工具**：棋盘工具栏提供「画地形」模式——画矩形/多边形即定义为阻碍地形（LOS 阻断、提供掩护源）。每块地形可标属性：`阻碍LOS` / `可站立(制高点)` / `飞蝇云(遮挡)` / `困难地形` / `可攀爬`。
- **掩护可视化**：选中特工时，引擎据 FR-9 实时计算并染色——特工控制范围内有阻碍地形 → 头像底圈绿色（有掩护）；2" 内有他特工 → 灰色（无掩护）。
- **遮挡可视化**：被飞蝇云等遮挡的目标用半透明虚线轮廓 + 「遮挡」标签，标明「阻止有效目标」。

### 7.3 可见性 / 射程 / 控制范围反馈

- **LOS 射线**：选中攻击方+悬停目标时，从头部到目标底座外缘画虚线，受阻段红色实线、通畅段绿色虚线。
- **射程环**：选中武器时，攻击方周围画武器射程半径环，环外目标置灰。
- **控制范围圈**：选中近战方/被近战威胁方时，画 1" 控制范围圈，圈内敌方高亮。
- **几何咨询式（D-17）**：引擎对每项几何（LOS/掩护/射程/控制范围）给出**可单击翻转的内联假设**；歧义项标「⚠ 可翻转」但**不弹框、不阻断**，默认套引擎值，玩家 1 击可翻；翻转后资格实时重算（§6）；所有判定入日志。

## 8. 可审计 UX

对应 FR-16（单步回滚）/ FR-17（日志回放）。这是信任的落点（UJ-2）。

### 8.1 流水线内的单步回滚

- 每个流水线步骤右侧有 `[◀ 回滚此步]`：回滚后该步及之后步骤变灰，回到该步的输入态（如允许重投/重录骰）。
- 回滚只影响会话内状态，不持久（D-20）。
- 回滚时棋盘状态也同步回退到该步发生前（伤害未应用、effect 未结算）。

### 8.2 日志回放面板

右侧（桌面）/底部抽屉（平板）的「历史」区，按时间倒序列出本局所有结算与回合动作：

```
日志回放                          [筛选: 全部/射击/近战/计谋/回合]
──────────────────────────────────────────────────
t6  射击 #3  步兵A→恐虐兵  0伤         [▶回放][↶回滚到此]
t5  计谋    钢铁光环  死亡天使         [▶回放]
t4  近战 #2  队长→奸奇术  2伤(残废)    [▶回放][↶回滚到此]
t3  激活    步兵A  交战  射击+转移     [▶回放]
...
──────────────────────────────────────────────────
```

- 点 `[▶回放]` 把对应结算的流水线重新展开，逐步重演（玩家手动点「下一步」），每步附当时骰值与修正。
- 点 `[↶回滚到此]` 把全局状态回退到该事件之前（会话内），用于「这步算错了想重来」。
- 筛选器快速定位争议点；UJ-2 的「秒平争论」靠这一步。

### 8.3 每步依据展开

每步的 `[▾依据]` 展开后形如：
```
④保留 (普通:1 关键:1)
   规则: ≥命中属性=普通成功，自然6=关键成功，自然1必败
   输入骰: [2,5,1,6]  命中属性: 3+(残暴+1关键)
   推导: 5≥3→普通✓ ; 6→关键✓ ; 1→必败✗ ; 2<3→失败✗
   来源: KT Lite 射击流水线 §保留 ; 残暴规则卡
```
含规则条目、输入数据、推导链、原文来源——满足「可审计」的全部要件。

## 10. 关键状态与反馈设计

### 10.1 人工裁定状态（护栏 / D-17 / OQ-4）

当引擎遇到**规则缺口**（协助 / 戒备 等 KT Lite 未定项），**不静默猜**，显式呈现（几何歧义改用 §7.3 咨询式翻转，不走此卡）：

- 受影响实体挂「⚠ 人工裁定」徽标。
- 行动指挥区出现一张裁定卡：
```
┌──────────────────────────────────────┐
│ ⚠  需人工裁定                         │
│ ─────────────────────────────────── │
│ 情形: （实现期发现的真·规则歧义）      │
│ 引擎无法自动判定该行动资格，交双方裁定。│
│ 请双方裁定:                           │
│   [允许]  [禁止]                      │
│ 记录理由(可选): [___________]         │
│ 此裁定将记入日志(可审计)。             │
└──────────────────────────────────────┘
```
裁定结果作为输入进入流水线，并在日志中标记为「人工裁定」来源——保持整局可审计、不破坏信任。

> **v1 范围说明（D-23）**：协助/戒备等已排除在 v1 范围外（Lite 无），此人工裁定卡**大概率不触发**，仅留作实现期发现真·规则歧义时的兜底。

### 10.2 Effect 到期

- 每个 effect 在单位卡片上显示剩余转折点数（如「毒 ×2TP」）。
- 推进到结束转折点时，引擎自动结算到期（push 提示「effect 到期 ×N」+ 列表），不静默移除。
- 到期事件记入日志。

### 10.3 受伤 / 受创 / 残废

| 状态 | 视觉 | 自动施加 |
|---|---|---|
| 健康 | 正常 | — |
| 受伤 (<起始) | 头像黄色描边 + 角标 | — |
| 受创 (<一半) | 头像橙色描边 + 「受创」标签 | 移动 -2"、武器命中 -1（引擎自动计入流水线修正） |
| 残废 (耐伤 0) | 头像灰阶 + ✕ | 移出棋盘（保留在面板灰显，便于日志回溯） |

阈值由引擎持续判定（FR-1），状态变化时单位卡片有 1 次脉动 + 日志记录。

### 10.4 主动玩家切换

- 顶部状态带整条换阵营色 + 文字「主动: XXX」。
- 整屏 2s 色带横扫（从旧色过渡到新色），不弹模态。
- 行动指挥区同时换色 + 更新 push 提示。
- 双方就绪/待机状态在单位面板同步刷新（对手的就绪特工亮起）。

## 11. 开局准备流程：地图选择 + 部署（G1 + G3）

建队完成 → 进入对局 → **选图** → **部署** → 转折点 1。这一段是对局主界面（§3）之前的前置流程。

### 11.1 地图选择（G3，D-28）
进入对局首屏 = 地图选择：
- **预设模板网格**：缩略图 + 名称（如「废墟交锋」「据点争夺」）；点击预览（地形/目标点/降落区），`[载入 ▶]`。
- **自定义画**入口：「空白板」→ 进画地形模式（§11.2）。
- 载入后展示完整棋盘（地形多边形着色 + 目标点图标 + 双方降落区色块）→ `[开始部署 ▶]`。

### 11.2 自定义画地形工具（G3 细节）
- 工具栏：`[矩形] [多边形] [橡皮] [复制] [镜像] [网格吸附]`；英寸标尺 + 网格背景辅助对齐物理板。
- 画完一块 → 弹属性标签（多选）：`阻碍LOS` / `掩护源` / `飞蝇云(遮挡)` / `困难地形` / `可攀爬` / `制高点`。
- 独立按钮放置**目标点**与双方**降落区**多边形。
- 自定义板**会话内有效，刷新重置**（D-20）；想复用须走预设模板（数据包发布）。

### 11.3 部署阶段（G1，FR-24）
- push：「轮到 A — 在降落区放置特工」。
- 交互：从单位面板**拖特工到己方降落区**（外红框拦截「须在降落区内」）；落子显示底座圆 + 朝向。
- **交替部署**：A 放 1 → B 放 1（push 切主动玩家 + 色带切换；D-19 全公开，无需遮罩）。
- 非法拦截：出降落区 / 与已放特工重叠 → 拦截卡（§6）。
- 全部部署完 → `[开始转折点 1 ▶]` → 进入对局主界面（§3）。

## 12. 目标点与得分可视化（G2，FR-25/26/27）

### 12.1 棋盘上的目标点
- 目标点 = 菱形图标 + 编号（OBJ1/OBJ2…）+ **控制范围圈**（虚线，`controlRange`）。
- **控制归属染色**：圈/图标角染控制方阵营色（A 蓝 / B 红 / 中立灰），随特工进出范围**实时更新**。
- 悬停 → tooltip「控制方: A（范围内 A×2 / B×1）」。

### 12.2 状态带 VP（FR-26）
- 状态带常驻：`VP  A:3  B:2  ·  转折点 2/4`。
- **转折点结束计分动效**：每控制 1 目标点 → 该方 VP +1（计数跳动 + 目标点图标脉冲）；push「TP 结束 — A 控制 2 点，VP +2」。

### 12.3 胜负（FR-27）
- 4 转折点结束 → 全屏结果页：`A 胜 — VP 8:5`（平局则标「平局」+ 规则要点）；`[再开一局]` = 刷新重置（D-20）。
- VP 变化可在日志回放（§8）按「计分」筛选查看。

**目标点 + VP 线框（对局态局部）：**
```

              ╲ 控制范围虚线圈(蓝)
            ◆OBJ2(灰·中立)

```

## 13. 设计原则之外的开放 UX 问题（留给后续）

- `[UX-OQ-1]` **骰源切换的颗粒度**：全局一个骰源开关，还是允许「这次射击用电子、那次用物理」？倾向后者（更贴现实混用），但需确认 UI 不混乱。
- `[UX-OQ-2]` **棋盘缩放/平移手势与拖放特工的冲突**（平板）：双指缩放 vs 单指拖特工需要明确的手势优先级与「拖动 vs 点击」的阈值（长按？位移阈值？）。需做可用性测试。
- `[UX-OQ-3]` **流水线默认展开 vs 折叠**：默认全展开（最透明但占屏）还是默认折叠结论行（最简洁但需点击查依据）？倾向默认折叠 + 高频步骤（投骰、伤亡）常驻可见。
- `[UX-OQ-4]` **多枚攻击骰/防御骰的并行展示**：一次投 4 颗攻击骰 + 3 颗防御骰时，如何同屏呈现而不混乱？倾向「攻击骰上排、防御骰下排、按成功/失败色编码」。
- `[UX-OQ-5]` **反应（Reaction）的时序打断 UX**：反应在对方激活中触发，如何既醒目又不打断对方节奏？需配合回合状态机的时序设计细化。
- `[UX-OQ-6]` **建队的「阵营子标识」选择时机**：是选阵营时立即选印记/战团战术，还是建队末尾统一选？影响认知负担。
- `[UX-OQ-7]` **横屏锁定的强制性**：平板竖屏是否完全禁用并提示「请横屏」？还是给一个降级的竖屏布局？倾向禁用并提示（降低维护面），但需确认。
- `[UX-OQ-8]` **规则查询浮层 vs 侧栏**：从结算流水线内联查规则时，是浮层（不离开上下文）还是跳转到规则区（更完整）？倾向浮层 + 「在规则区打开」二级入口。
- `[UX-OQ-9]` **可视化反馈的性能边界**：实时 LOS 射线 + 射程环 + 控制范围圈在密集棋盘上的帧率，需在原型阶段验证，必要时降级为「按需显示」。
- `[UX-OQ-10]` **配色与色盲安全的具体方案**：阵营分色需通过色盲模拟（红绿/蓝黄），待视觉阶段定稿。
