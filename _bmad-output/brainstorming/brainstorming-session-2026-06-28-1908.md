## Phase 1: Five Whys —— 根因闭环

**现象：** 对局中算错/卡壳。

**链：**
```
Why1  崩在多来源结算点（资格×条件×叠加×骰子）   ← 用户 4 个现场案例
Why2  4 层规则散落成平铺文字，脑里临时拼流水线
Why3  必崩：工作记忆溢出(a) + 叠加隐性(b) + 状态漂移(c)
Why4  桌上无载体同时持有「状态+effect栈+规则」并计算
Why5  根：桌面缺一个「结算引擎」  ← 产品锚点
```

**根因结论（产品定位锚）：** 助手的根价值**不是规则查询**，是把结算流水线做成可计算状态机——持有「单位当前状态 + 生效 effect 栈 + 规则」，在每个结算点算出：能不能做 / 怎么修正 / 谁先叠 / 投几颗。规则查询与掷骰辅助是这个引擎的「面」。

### 浮现的引擎子系统（现场案例 → 功能面）

- **[伤害结算器]**（脊椎）——完整伤害流水线，分**射击线**与**近战线**：命中 → 致伤/暴击 → 防护(掩体/护甲) → 减伤/附魔 → 不屈豁免 → 伤害 → 伤亡
- **[资格判断器]** 案1 → 可见性×掩体 → 能否射击 + 伤害修正
- **[行动状态机]** 案2 → 撤退后合法动作集
- **[effect 栈解析器]** 案3 → 跨来源 buff/debuff 叠加顺序与互斥（喂进伤害结算）
- **[骰子推导器]** 案4 → 特殊豁免(不屈)触发时机 + 数量反推

## Phase 2: SCAMPER —— 功能爆破（锁定点子）

用户每轮直觉挑选，下面为**锁定**项（未选种子略）。骰源结论：**电子投 或 物理骰录入两可**，都自动套修正。

**S · Substitute（替换）**
- **[S1] 状态托管** —— HP/生效 effect/行动状态/装备由 app 持续持有，回合自动到期 effect，结算点状态现成。
- **[S2] 电子骰 + 自动修正（骰源无关）** —— 投骰池后自动叠命中/致伤/防护修正，一键走完整流水线，分射击/近战预设；物理骰可手投后录入。

**C · Combine（合并）**
- **[C1] 引擎本体** —— 状态+骰子+规则合成一条结算流（=Why#5 的结算引擎）。
- **[C2] 「我打这个目标」一击交互** —— 点目标 → 资格(可见/射程/掩体) → 自动接伤害流水线 → 出伤亡，不切屏。
- **[C4] 棋盘双向镜像** —— 数字面 = 物理棋盘实时快照，位置/状态双向同步（同步方式待架构定）。

**A · Adapt（改造）**
- **[A1] effect 条件系统（借 RPG VTT）** —— effect 当图标+计时器挂单位，回合结束自动到期，点图标看来源。

**M · Modify（修改）**
- **[M2] 单步 + 回滚** —— 结算可下一步/暂停/退回，也能整回合批量结，算错可逆。
- **[M3] 结算日志/回放** —— 全程留痕可回看「这步怎么算的」，专治规则争议（信任层）。

**P · Put to other use（挪用）**
- **[P1] 规则可插拔** —— 规则当数据包加载，引擎复用到完整版/新阵营/扩展（架构须 data-driven）。
- **[P4] 远程对局 / 裁判工具** —— 状态联网同步，异地也能打 / 当第三方裁判。

**E · Eliminate（删减）**
- **[E1] 删「翻规则书」** / **[E2] 删「手算叠加」** / **[E3] 删「纸笔记血量/effect」** / **[E5] 删「裁判负担（谁激活过/该谁动）」** —— 把对局内全部认知负荷移走。
- **[E6] MVP 切边** —— 战役追踪、教学层**后置**，v1 纯对局内结算。
- ⚠️ **保留物理骰**（E4 被否）→ 骰源两可。

**R · Reverse（反转）**
- **[R1] push 式交互** —— 引擎主动推「该结算/该谁动」，玩家只确认。
- **[R2] 先验合法性** —— 结算前先查「合法吗/缺什么」，算前拦截。
- **[R4] 对方视角** —— 切对手视角看威胁/可见性，战术辅助。

### SCAMPER 浮现的产品轮廓（v1）

**结算引擎核心** + **push 式一击交互（先验合法性）** + **多端棋盘双向镜像（远程/裁判）** + **可插拔规则** + **全程可审计（单步回滚+日志）** + **去载化（删书/手算/纸笔/裁判负担）**。范围：对局内；战役/教学/AI对手后置。

### 关键架构信号（喂后续架构阶段）
- 引擎须 **data-driven**（规则可插拔，P1）
- **棋盘同步：已消除**（不做联网）；平台 = 静态 Web + GitHub Pages，无后端
- **信任/可审计**是非功能刚需（M2/M3）
- **骰源两可**：结算逻辑与骰子来源解耦

## 锁定的产品决策

- **平台/部署**：静态 Web 应用，GitHub Pages，无后端
- **联网**：OFF（无实时同步）
- **多人交互**：**单屏共用**（pass-and-play / 一人操作）
- **跨设备使用**：存档导入导出 / 分享码
- **骰源**：两可（电子投 或 物理骰录入），都自动套修正
- **范围**：v1 = 对局内结算引擎；战役/教学/AI 对手/远程 后置
- **架构原则**：规则 data-driven（JSON 数据包），引擎与骰源解耦，结算全程可审计

## Session Summary and Insights

- **根因（Five Whys）**：桌面缺结算引擎 → 助手核心 = 可计算状态机；规则查询/掷骰只是其"面"。
- **差异化锚点**：现存工具（建队器/沙盒）无人拥有对局内结算流水线。
- **范围纪律**：一路砍 AI 对手/教学/战役/联网，聚焦"对局内结算 + 单屏共用 Web"，风险大降。
- **隐形成本**：3 阵营规则数据建模是最大工作量；可审计性是采纳前提。
